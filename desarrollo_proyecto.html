2. Metodología de desarrollo

Aunque el equipo es individual, se han adoptado prácticas inspiradas en Scrum/Kanban ligero:

  • Iteraciones semanales con objetivo único (ej: "auth + perfiles", "carrito + checkout", "panel admin estadísticas").
  • Tablero de tareas en GitHub Issues (To do / In progress / Done).
  • Commits atómicos con mensaje convencional (feat:, fix:, docs:, refactor:).
  • Pruebas tras cada feature: análisis estático, prueba manual web y Android, validación de RLS en Supabase.
  • Tutorías semanales con el profesor como ceremonia de review.

Principio guía: "primero funcional, luego pulido". Cada feature se ha cerrado con una versión navegable antes de pasar a la siguiente, evitando estados a medio terminar.

5. Decisiones técnicas clave

5.1 Riverpod 2 con codegen

Se eligió Riverpod sobre alternativas (Provider, Bloc) por su tipado fuerte, su capacidad de invalidar dependencias granulares y la generación automática con @riverpod, que elimina el boilerplate de declarar manualmente cada provider:

@riverpod
Future<List<Dish>> menuDishes(MenuDishesRef ref) async {
  return ref.read(menuRepositoryProvider).list();
}

// Uso en widgets:
final dishes = ref.watch(menuDishesProvider);
dishes.when(
  data: (list) => DishGrid(items: list),
  loading: () => const LoadingIndicator(),
  error: (e, _) => ErrorView(error: e),
);

5.2 Freezed para los modelos de dominio

Todos los modelos de dominio son inmutables con @freezed y serializables JSON con fieldRename: FieldRename.snake para casar con las columnas Postgres:

@freezed
class Order with _$Order {
  const factory Order({
    required String id,
    @JsonKey(name: 'user_id') required String userId,
    @JsonKey(name: 'order_type') required OrderType orderType,
    required OrderStatus status,
    required double total,
    @JsonKey(name: 'created_at') required DateTime createdAt,
  }) = _Order;

  factory Order.fromJson(Map<String, dynamic> json) => _$OrderFromJson(json);
}

5.3 GoRouter con ShellRoute

Una única definición de rutas para Web (URL hash) y Android. Se usa ShellRoute para mantener la navegación principal (tabs) y rutas anidadas con redirecciones para guards de autenticación y rol.

5.4 Supabase Realtime para el estado del pedido

El detalle de pedido se suscribe vía supabase.channel('orders:<id>').onPostgresChanges(...) y refresca el provider sin polling. El cliente ve los cambios de estado (preparando → listo → en reparto → entregado) en tiempo real.

5.5 RLS como única fuente de autorización

Toda la seguridad se delega a las políticas RLS de Postgres. El cliente Flutter nunca filtra por user_id manualmente: Supabase lo hace por nosotros vía la sesión JWT. Esto elimina toda clase de bypass por manipulación de cliente.

5.6 Edge Functions para integraciones críticas

Las claves secretas (Stripe, Brevo, FCM) nunca viven en el cliente. Se usan 8 Edge Functions en Deno como proxy seguro: create-payment-intent, send-order-notification, send-encargo-confirmation, send-encargo-reminders, send-catering-notification, send-newsletter, send-welcome-email, chat-bot.

5.7 Pagos cross-platform

Stripe se integra con flutter_stripe en Android (Payment Sheet) y con stripe_js en Web (redirect + Webhooks). El switch se hace con kIsWeb en la capa de checkout.

7.2 Pruebas unitarias y de widget

El directorio test/ contiene pruebas para los repositorios críticos y para componentes de UI compartidos. Ejemplo: test/features/auth/data/repositories/auth_repository_test.dart verifica el flujo de login, refresh de sesión y manejo de errores.

7.3 Pruebas manuales por rol

Para cada rol se ha ejecutado un checklist end-to-end:

  • Cliente: registro → catálogo → carrito → pago → recibir notificación push → ver pedido en realtime → valoración.
  • Empleado cocina: ver cola de pedidos → marcar en preparación → marcar listo → escanear QR del cliente.
  • Empleado repartidor: ver asignados → mapa → marcar entregado → liberar pedido.
  • Admin: crear plato → activar oferta → revisar dashboard → responder chat soporte → enviar newsletter → publicar reseña.

7.4 Pruebas de integración con Supabase

  • Validación de políticas RLS por rol (cliente no puede leer pedidos ajenos, empleado no puede modificar precios, admin tiene acceso total).
  • Disparo de triggers (creación de perfil en handle_new_user, actualización de updated_at).
  • Verificación de constraints (unicidad de email en newsletter, claves foráneas en order_items).

8. Generación de código

Tras cualquier cambio en modelos Freezed o providers Riverpod hay que regenerar:

dart run build_runner build --delete-conflicting-outputs

Si el codegen falla con "Cannot recurse at later or equal phase", ejecutar primero un clean:

dart run build_runner clean
dart run build_runner build --delete-conflicting-outputs
