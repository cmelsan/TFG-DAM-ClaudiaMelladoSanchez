1. Introducción al diseño

Este documento describe la arquitectura técnica, el modelo de datos y los flujos de interacción de la aplicación multiplataforma para la gestión integral de un negocio de comida preparada para llevar. El sistema cubre cinco líneas de servicio: venta en mostrador, encargos programados, reparto a domicilio, recogida en local y catering de eventos, además de un canal de contacto para consultas generales, propuestas de colaboración y ofertas de empleo.

Principio de diseño: navegación libre sin registro. El usuario puede consultar el menú completo, usar el chatbot IA y añadir al carrito sin crear cuenta. El registro solo se solicita al tramitar el pedido.

2. Arquitectura del sistema

La aplicación sigue una arquitectura cliente-servidor con un frontend Flutter multiplataforma (Android + Web) que se comunica con Supabase como Backend-as-a-Service. Los servicios externos (pagos, notificaciones, emails, IA) se integran mediante Edge Functions serverless.

9. Decisiones de diseño y patrones

9.1 Arquitectura de software

PatrónAplicaciónJustificación
Feature-firstEstructura de carpetasCada feature (auth, orders, menu, catering, chat...) es un módulo autónomo con su UI, estado y datos
Repository patternAcceso a datosCapa intermedia entre Riverpod y Supabase. Facilita testing y cambio de fuente de datos

9.3 Seguridad

  • Autenticación: JWT de Supabase, tokens persistentes, refresh automático
  • Autorización: RLS en todas las tablas. Un cliente solo ve sus pedidos, un empleado ve todos los pedidos pero no puede modificar platos, etc.
  • Pagos: El Payment Intent se crea en una Edge Function (servidor), nunca en el cliente. El client_secret se usa solo para confirmar el pago
  • API keys: Las claves de Gemini, Stripe secret y Brevo se almacenan en las variables de entorno de Supabase Edge Functions, nunca en el código del cliente
  • Validación: Doble validación — en el frontend (UX) y en la BD (constraints + triggers)
  • Imágenes: Compresión antes de subir (flutter_image_compress). Bucket público para platos, privado para documentos

9.4 Rendimiento

  • Caché local: Carrito en SharedPreferences/localStorage (sin necesidad de red)
  • Realtime selectivo: Solo se suscriben a Realtime las pantallas que lo necesitan (KDS, estado pedido, plato del día, pedido grupal)
  • Lazy loading: Imágenes de platos con CachedNetworkImage y placeholders
  • Paginación: Listados de pedidos y platos con scroll infinito
