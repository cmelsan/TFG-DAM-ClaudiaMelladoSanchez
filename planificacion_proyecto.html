1. Objetivo del documento

Este documento recoge la planificación detallada de la ejecución del Proyecto Final DAM "Sabor de Casa", una aplicación multiplataforma (Android + Web) para la gestión integral de un negocio de comida preparada y catering. Complementa los entregables previos (Análisis del Contexto y Diseño del Proyecto) y constituye el plan operativo que ha guiado el desarrollo durante 11 semanas.

Resultado: aplicación funcional desplegada, 28 migraciones SQL aplicadas, 8 Edge Functions activas, 41 pantallas implementadas en 3 perfiles de usuario (cliente / empleado / administrador), build web sin errores y APK Android firmado.

2. Secuenciación de tareas y plazos

El proyecto se ha desglosado siguiendo la guía oficial DAM en 5 grandes fases más la defensa. Cada fase produce uno o varios entregables verificables.

Fase 1 · Análisis del contexto (sem. 1-2)

Estudio del sector hostelería, identificación del problema, entrevistas informales, análisis DAFO y entregable analisis_contexto.html.

Fase 2 · Diseño del proyecto (sem. 3-4)

Requisitos funcionales y no funcionales, arquitectura por capas, modelo entidad-relación, mockups de las 41 pantallas y entregable diseno_proyecto.html.

Fase 3 · Planificación (sem. 5)

Diagrama de Gantt, asignación de recursos, análisis de riesgos y plan de contingencia (este documento).

Fase 4 · Desarrollo e implementación (sem. 6-9)

Scaffolding Flutter, configuración Supabase, autenticación, módulos cliente, módulos empleado, panel admin, Edge Functions, integración Stripe, pruebas funcionales.

Fase 5 · Seguimiento, evaluación y documentación (sem. 10)

Manuales técnico y de usuario, registro de incidencias, evaluación, generación del paquete final.

Defensa del proyecto (sem. 11)

Presentación oral, demostración en vivo de la aplicación y resolución de dudas del tribunal.

5. Análisis de riesgos y plan de contingencia

Se ha realizado un análisis cualitativo de los principales riesgos del proyecto, evaluando probabilidad e impacto en una escala baja / media / alta. Para cada riesgo se define una medida preventiva y, si se materializa, un plan de contingencia.

IDRiesgoProb.ImpactoMitigación / contingencia
R1 Curva de aprendizaje de Flutter + Riverpod + Freezed superior a lo previsto. Media Alto Spike previo de 3 días con plantilla mínima; uso de codegen para reducir boilerplate; documentación oficial y skills internos.
R2 Errores en las migraciones SQL aplicadas a Supabase (orden, RLS, claves). Media Alto Numeración estricta de migraciones, backup antes de cambios de esquema, aplicación manual en Studio SQL Editor cuando db push falla.
R3 Bloqueo de pagos en producción por requisitos de Stripe (SCA, métodos europeos). Baja Alto Trabajar todo en modo Test; documentar el switch a Live solo en deploy final; usar Payment Sheet (móvil) y Stripe.js (web) ya certificados.
R4 Notificaciones push (FCM) no entregadas en algunos dispositivos. Media Medio Doble canal: push + email Brevo paralelo. Reintentos best-effort en la Edge Function. Logs de entrega en panel admin.
R5 Diferencias de comportamiento entre Web y Android (router, layouts, Stripe). Alta Medio Test manual en ambas plataformas tras cada feature, condicionales kIsWeb donde aplica, ShellRoute compartido para evitar divergencia.
R6 Pérdida de datos del repositorio o del proyecto local. Baja Alto Push diario a GitHub; backup adicional semanal a disco externo; uso de la rama main como única fuente de verdad.
R7 Codificación UTF-8 corrupta al editar archivos con acentos desde PowerShell. Media Medio Editar siempre desde VS Code; nunca Add-Content @"..."@ con caracteres acentuados; recuperación desde historial local de VS Code en caso de corrupción.
R8 Cuota agotada en plan Free de Supabase / Brevo en producción. Baja Medio Monitorizar uso semanal; plan B documentado con upgrade a planes Pro mensuales.
R9 Retraso en la fase de pruebas por acumulación de features. Media Medio Pruebas continuas tras cada feature, no acumular QA al final; widget tests para los componentes críticos.
R10 Imposibilidad de defensa en fecha por motivos personales o de salud. Baja Alto Adelantar entrega 1 semana respecto al deadline oficial; tener slides y demo grabada como backup.

7. Indicadores de seguimiento

Durante la ejecución se han usado los siguientes indicadores para verificar el avance:

  • % de pantallas implementadas sobre las 41 previstas (objetivo final: 100 %).
  • Migraciones SQL aplicadas / planificadas (objetivo: 28 / 28).
  • Errores de dart analyze (objetivo: 0 errores; los info cosméticos son aceptables).
  • Build web sin wasm dry-run verde tras cada feature mayor.
  • Cobertura de roles: cliente, empleado y admin con flujos completos extremo a extremo.

Resultado a fecha de entrega: 41/41 pantallas, 28/28 migraciones, 0 errores de análisis, build web OK (≈ 20 MB), 3 roles funcionales, 8 Edge Functions desplegadas. El cronograma se ha cumplido sin desviaciones críticas.
